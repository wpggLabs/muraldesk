# MuralDesk

## Overview
MuralDesk is a local-first, visual desktop pinboard application built with React and Vite. It allows users to pin various media types (images, videos, notes, links) onto a canvas-style board. Key features include drag, resize, duplicate, and lock functionalities for pinned items. All user data, including layout and media binaries, is automatically saved locally on the user's machine, ensuring privacy and offline access. The project aims to provide a highly interactive and persistent digital workspace without relying on any backend, authentication, or cloud services.

## System Architecture
MuralDesk is a frontend-only application using React and Vite, designed for static deployment.
The UI/UX is designed as a full-viewport canvas where items render directly. Items appear as free-floating objects without permanent borders or headers, revealing controls (outline, mini-toolbar for lock/duplicate/delete) only on hover/select. A floating pill toolbar is located at the top-center for main actions (add items, export, import, fullscreen) and auto-dims when not in use.

**Key Technical Implementations:**
- **Drag and Resize:** Implemented using `react-rnd`.
- **Local Persistence:** `localStorage` is used for layout data (positions, sizes, text, metadata, lock status, z-index), while `IndexedDB` stores image and video binary large objects (blobs). Object URLs are revoked to prevent memory leaks.
- **PWA Capabilities:** Includes a web manifest and a basic app-shell service worker for installability and offline access, registering only in production builds.
- **Card Types:**
    - **ImageCard:** Displays images.
    - **VideoCard:** Plays videos with hover-only mute/loop controls.
    - **NoteCard:** Editable text notes with hover-only color swatches.
    - **LinkCard:** Smartly classifies and renders links as YouTube embeds, videos, images, web previews (favicon, title, description), or an "unsafe URL" placeholder. YouTube and video links have `pointer-events:none` by default for easy dragging, with an "Interact" mode to enable native player controls.
- **Selection & Interaction:** Items can be selected with a click, showing an accent outline. Keyboard shortcuts like `Esc` to deselect and `Delete`/`Backspace` to remove are supported.
- **Desktop Target (Electron):** An optional Electron shell provides a desktop application experience, sharing the same renderer. It includes a "Desktop Canvas Mode" for an immersive, full-screen experience and persists window state (position, size) locally. IPC between the renderer and main process is minimal and secured with `contextIsolation` and `sandbox` enabled.
    - **Transparent overlay mode (Electron only):** the Electron build creates its `BrowserWindow` with `transparent: true`, `frame: false`, `hasShadow: false`, `backgroundColor: '#00000000'`. The renderer flips a `data-electron="true"` attribute on `<html>` so `src/index.css` paints `html / body / #root` transparent, App.jsx skips its dark canvas background / ambient gradient / grid / empty-state hero, and only the floating toolbar pill plus pinned items remain visible over the user's desktop. The web/PWA build is untouched and keeps the original dark canvas. The toolbar pill is also the OS window-drag handle (`-webkit-app-region: drag`) since the frame is gone; each button opts out with `no-drag`. Frameless-window controls (minimize / close) are exposed via `window.muraldesk.minimizeWindow()` / `closeWindow()` (IPC channels `window:minimize` / `window:close`) and shown as compact buttons at the right end of the toolbar in Electron.
    - **Desktop Canvas Mode (Electron, full-display overlay):** entering Desktop Mode does NOT use `BrowserWindow.setFullScreen()` (OS-level fullscreen was the wrong fit for a transparent overlay — animation, taskbar hide, dedicated space on macOS, etc.). Instead the main process picks the display the window is currently on via `screen.getDisplayMatching(win.getBounds())`, captures the pre-overlay rect (and `isMaximized` flag) in memory only, then calls `win.setBounds(display.bounds)` so the BrowserWindow expands to cover the entire physical monitor. Pinned `<Rnd bounds="parent">` items can therefore be dragged anywhere across the whole display. The window stays a regular, non-fullscreen, transparent window so click-through, frame-less mode, and the toolbar all keep working. While in overlay we call `setMovable(false)` + `setResizable(false)` so the toolbar's drag region can never accidentally push the overlay off-screen, and `persistGeometry()` is gated on `!inOverlayDisplay` so the full-display rect never overwrites the saved normal geometry in `window-state.json`. Exiting Desktop Mode restores `setMovable/setResizable`, calls `setBounds(prevWindowBounds)`, and re-maximizes if we were maximized before — falling back to a 1280×800 window centered on the current display if the prev bounds are somehow missing. The renderer drives this through the existing `desktop:set-fullscreen` / `desktop:toggle-fullscreen` IPC channels (names kept for renderer compat); the main process emits `desktop:fullscreen-changed` directly from `enterOverlayDisplay` / `exitOverlayDisplay`. The OS-level `enter-full-screen` / `leave-full-screen` listeners were intentionally removed so OS fullscreen (if triggered externally) can't desync the renderer's Desktop Mode toggle from the real overlay state — `inOverlayDisplay` is the single source of truth. Module-level overlay state (`inOverlayDisplay`, `prevWindowBounds`, `prevMaximized`) is reset both at the top of `createWindow()` and inside `win.on('closed')` so a macOS dock-reactivation that calls `createWindow()` again after the previous window was destroyed in overlay mode starts cleanly.
    - **Electron toolbar cleanup:** in the Electron build the toolbar shows only the Desktop Mode toggle (the Fullscreen button is web/PWA-only) — the two were doing the same thing in Electron after the overlay rework, so the redundant Fullscreen pill was hidden to keep the toolbar feeling like a clean mural layer rather than a busy app shell. The web build still shows Fullscreen so the Ctrl/Cmd+Shift+F shortcut has its visible counterpart.
    - **Centered spawn in Desktop Mode:** when adding an image / video / note / link while `desktopMode` is true, the new item spawns near the center of the current viewport (= the full display, since the BrowserWindow covers it in overlay mode) with a small random jitter on `x` (±40px) and `y` (±30px). The default top-left jitter (80 + random*200) is kept for normal-window Electron and the web build because in those modes the viewport is small and centered spawning would feel cramped. Spawn coords are clamped to a minimum of 20px to ensure the card's drag handle is always visible.
    - **Sample board minimal mode (Desktop Mode only):** `buildSampleItems({ minimal: desktopMode })` returns just three clean mural objects (one image, one link, one small sticky note) centered on the viewport when called from Desktop Mode — no Welcome / Tips note panels. Web/PWA and normal-window Electron keep the original richer four-item set with onboarding Notes. Caller is `App.handleSampleBoard`.
    - **Toolbar visibility (Electron):** force-shown when the board is empty, then auto-hides once the first item exists. The toolbar starts HIDDEN on mount when `(isElectron || desktopMode) && hasItems` so launching MuralDesk into a session with existing pinned items doesn't flash the toolbar visible — the previous default of `nearTop=true` / `revealed=true` made the toolbar appear "stay visible because the app is focused" until the first mousemove fired. Re-appears immediately when the cursor enters the top 80 px reveal zone, the toolbar is hovered, any pinned item is hovered (App lifts hover state from `BoardItem.onHoverChange`), or a dialog is open. When all those triggers go false a 700 ms grace timer starts; if any becomes true again before it expires the timer is cancelled, so the toolbar never flickers and never "slips away" while the user moves toward it. Desktop Canvas Mode (OS-fullscreen) uses a tighter 24 px band so the toolbar doesn't constantly flash back on. The web build retains its original always-present-but-dimming toolbar (110 px reveal band).
    - **Click-through (Electron only):** transparent empty areas of the window do not block desktop clicks. A renderer hook (`src/hooks/useElectronClickThrough.js`) watches `mousemove` + `mouseup` and uses `document.elementFromPoint` to detect whether the cursor is over an interactive zone (`[data-muraldesk-interactive]` on the toolbar / dialog, `.muraldesk-card-rnd` on each pinned card). It calls `window.muraldesk.setIgnoreMouseEvents(ignore, { forward: true })` (IPC channel `window:set-ignore-mouse-events`) whenever that state changes. **Initial state is set by a real hit-test, not a guess:** on mount the hook calls `window.muraldesk.getCursorPositionInWindow()` (IPC channel `window:get-cursor-position-in-window`, backed by `screen.getCursorScreenPoint()` minus the window's content bounds) to get the OS cursor position translated into renderer client coords, then runs `elementFromPoint` at that position before any mousemove arrives — so the very first click after launch is correct whether the cursor is over an empty area (passes through to desktop) or over the toolbar/a card (interactive). `forward: true` keeps `mousemove` events flowing while clicks pass through, so the renderer can flip click-through back off the instant the cursor enters a card or the toolbar. While any mouse button is held (`e.buttons !== 0`) the hit-test is skipped entirely so an in-flight drag/resize gesture from react-rnd or a window-drag from the toolbar can never be aborted by a mid-gesture click-through flip; `mouseup` re-runs the hit-test immediately so click-through engages without waiting for the next idle mousemove. On unmount / HMR the hook restores `setIgnoreMouseEvents(false)` so a hot-reload doesn't orphan the window in click-through. Both IPC handlers resolve the target window from `event.sender`; the set-ignore handler only honors `{ forward: true }` (other option keys are stripped); the cursor-position handler returns `null` (rather than throwing) if the window is destroyed. Web build has no bridge, hook is a no-op.
    - **Full-window canvas:** `index.css` sets `html, body, #root { width: 100%; height: 100%; overflow: hidden }`, App.jsx renders its root div as `width: 100vw; height: 100vh; position: relative; overflow: hidden`, and every pinned `<Rnd>` uses `bounds="parent"`. There is no centered app container, no max-width, no padding, no margin — items can be dragged and resized anywhere inside the visible MuralDesk window. Positions are persisted to `localStorage` via the existing `useBoard` hook on every change.
    - **Dev ports:** the web/PWA dev server runs on **5000** (`npm run dev` / `dev:web`, host `0.0.0.0`, used by the Replit preview). The Electron renderer has its own dedicated dev server on **5173** with `--strictPort` (`npm run dev:renderer`), so it can never silently drift to a different port if 5000 is busy. `npm run dev:desktop` runs `dev:renderer` and starts Electron with `ELECTRON_RENDERER_URL=http://localhost:5173`.
    - **Replit limitation:** Electron itself cannot launch in the default Replit container — Chromium needs GUI system libraries (GTK/GLib: `libgobject-2.0.so.0`, `libgtk-3.so.0`, …) that are not installed here, so `npm run dev:desktop` and `npm run build:desktop` will fail with a `cannot open shared object file` error. Use Replit for editing and for the web build (`npm run dev:web` / `build:web`); test the Electron shell on a Windows / macOS / Linux desktop machine where those libraries are standard. Linux desktops without a compositing window manager will render the "transparent" window opaque; Windows 10/11 and macOS render it correctly.

**Features:**
- **Sample Board:** Quick creation of a demo board with pre-filled cards.
- **Duplicate:** Clones any card, including copying media blobs for media items.
- **Lock:** Prevents dragging and resizing of cards.
- **Export/Import JSON:** Allows saving and loading board layouts, with import validation and URL sanitization.

## External Dependencies
- **React:** JavaScript library for building user interfaces.
- **Vite:** Next-generation frontend tooling for fast development.
- **react-rnd:** A React component for draggable and resizable elements.
- **localStorage:** Web Storage API for local data persistence.
- **IndexedDB:** Low-level API for client-side storage of large amounts of structured data.
- **Electron:** Framework for building desktop applications with web technologies (optional, for desktop target).
